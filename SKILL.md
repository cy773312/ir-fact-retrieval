# IR Fact Retrieval

## Overview

把用户的问题转化为一份可研究、可引用、可继续深挖的事实底稿。
先界定事件边界，再检索高可信来源，最后按“已确认 / 主流认定 / 存在争议 / 待核实”组织信息。
本 skill 不依赖额外的 `scripts/`、`references/` 或 `assets/`；默认仅按本文件执行。

## Workflow

每一步都要产出一个可复用的中间结果：

- Step 1 产出：事件边界定义（事件名、时间边界、地区、主要行为者）。
- Step 2 产出：来源清单（按官方/权威机构/主流二手资料分组）。
- Step 3 产出：事实分级清单（每条核心事实附可信度标签）。
- Step 4-6 产出：可直接写入报告的背景、行为者、争议要点。
- Step 7 产出：与用户语言一致的最终结构化报告。

默认按 Step 1 → Step 7 顺序执行；只有当 Step 2 发现事件边界设错或过宽时，才回到 Step 1 重新界定。

### 1. Define the event boundary

先把问题限定到一个可检索对象：

- 识别事件名称、地区、时间范围、主要行为者。
- 判断用户问的是单一事件、持续议题，还是某一轮外交过程。
- 若用户表述模糊，先在内部做最小必要限定；必要时再说明你的限定口径。

### 1.5 Checkpoint before deep retrieval

在以下情况，先暂停并确认一次，再继续检索：

- 用户同时混合多个事件、多个时间段或多个地区。
- 用户把“事实检索”与“责任判断/政策建议”混在一起。
- 事件仍在快速演变，且时间口径会显著影响结论。

确认时只问最小必要问题；如果不值得打断用户，则明确写出你采用的检索口径与时间边界，再继续。
确认问题默认压缩为一句，优先确认：

- 你要我聚焦哪个具体事件/时间段？
- 你要事实底稿，还是还要责任判断/政策分析？

用户一旦给出确认，后续报告必须严格沿用该口径；若未确认，则在开头先声明你代为采用的口径。

### 2. Research before answering

这类问题高度依赖时效性与来源质量。默认先检索，再回答。

优先级从高到低：

1. 官方与原始文件
- 条约、协定、联合国决议、国际法院或仲裁文本。
- 各国政府、外交部、国防部、总统府、议会、官方通报。
- 国际组织：UN、ICJ、ICC、NATO、EU、OSCE、IMF、World Bank、UNHCR、OCHA 等。

2. 主流机构与权威数据库
- SIPRI、ICRC、World Bank、IMF、UCDP、ACLED、Our World in Data 等。
- 仅在其与问题直接相关时使用。

3. 主流媒体与学术二手资料
- Reuters、AP、BBC、Financial Times、The Economist 等用于时间线核对。
- 学术机构、智库、综述文章用于“主流认定”与解释性框架。

避免把单一媒体报道、匿名消息、社交媒体帖子直接当作已确认事实。

### 3. Grade every claim by confidence

对关键事实进行显式分级：

- `已确认`：有官方文件、条约原文、法院裁决或多方一致记录支持。
- `主流认定`：学术界、主要国际组织或多家高质量来源形成较稳定共识。
- `存在争议`：行为者之间、媒体之间或研究者之间存在明显冲突说法。
- `待核实`：仅见单一来源、证据链不完整、或数据尚未稳定。

规则：

- 事实与解释分开标记。
- 数字必须带时间点，例如“截至 2026 年 4 月”。
- 若不同来源给出不同数字，优先并列呈现并解释口径差异，不要强行合并。

### 4. Add only necessary structural context

主动补足理解该事件所必需的背景，但不要扩展成泛泛地区史。

优先补充：

- 历史前置条件：边界形成、停火线、前次战争、殖民遗产、政权更替。
- 制度约束：国际法、安理会决议、同盟义务、停火机制、观察团、仲裁或法院程序。
- 关键前置事件：导致当前升级或谈判的直接节点。

不要加入与问题无直接关系的宏大叙事。

### 5. Distinguish actors, interests, and actions

至少拆出：

- 主要行为者：国家、国际组织、武装组织、代理人、调停方。
- 核心利益：领土、安全、联盟信誉、国内政治、能源、贸易、人道压力等。
- 公开立场：正式声明、法律主张、外交叙事。
- 实际行动：军事部署、制裁、投票、谈判、承认、撤军、军援、封锁等。

如公开立场与实际行动存在落差，明确写出。

### 6. Surface disputes without taking sides

冲突类议题至少识别两类分歧：

- 事实争议：边界线、伤亡数字、开火顺序、是否使用某种武器、是否违反协议。
- 解释争议：责任归属、法律定性、是否构成侵略、是否属于自卫、是否违反国际人道法。

用中性表述，不替任何一方背书。

### 7. Write in the user's language

- 用户用中文提问，就用中文输出。
- 用户用英文提问，就用英文输出。
- 用户用其他语言提问，尽量用同一语言输出；若资料主要来自英文且翻译存在不确定性，可简短说明。

## Output Template

按事件重要性灵活调整篇幅，但默认使用以下结构：

### 一、事件概览

- 事件的核心定义。
- 时间跨度与当前状态。
- 地理范围与主要涉事国/组织。

### 二、结构性背景

- 理解该事件所必需的历史前提。
- 相关制度框架。
- 关键前置事件。

### 三、行为者分析

用表格或分条列出：

- 主要行为者
- 各方核心利益
- 各方公开立场
- 各方实际行动

### 四、关键事件时间线

按时间顺序列出转折节点，每条包含：

- 时间
- 事件
- 直接影响或意义

### 五、核心争议与分歧

- 事实层面的争议
- 解释层面的争议
- 各方立场差异

### 六、关键数据与文件

- 重要量化数据，注明来源与时间点
- 关键条约、协议、决议、裁决或声明的名称与核心内容摘要

### 七、延伸关联

- 与该事件高度相关的平行议题或关联事件
- 对地区秩序、联盟结构、国际规范或后续危机的潜在影响

### 信息缺口

在以下情况必须补充本节：

- 关键数据无法交叉验证
- 用户问到的时间点非常新，公开信息仍未稳定
- 某些说法只有单一来源
- 缺少官方文本、原始决议或可确认统计

指出还应进一步查阅什么，而不是笼统地说“信息不足”。

## Formatting Rules

- 使用 Markdown。
- 对存在争议或证据不足的信息，在句末标注：`（存在争议）`、`（待核实）`。
- 关键数据必须带时间点。
- 尽量把“事实”与“解释”分开写，避免混在同一句里。
- 如果已经浏览并检索，尽量附上来源链接或来源名称，尤其是官方文件与关键数据。
- 来源标注优先使用 `来源名称 + 日期`，例如：`BIS，2024-12-02`、`中国外交部，2025-01-14`；若有链接，再附链接。
- 不做价值判断，不替用户下政策结论，除非用户另有要求。

## Default Response Pattern

默认先给一段极简导语，说明检索口径，例如：

> 以下内容按“已确认 / 主流认定 / 存在争议 / 待核实”整理，重点是帮助建立事实底稿，而非提供规范判断。

随后进入正式结构化报告。

出稿前快速自检：

- 是否先回答了“这是什么、何时开始、目前到哪一步”。
- 是否把“已确认事实”和“存在争议的解释”分开了。
- 是否至少给出一个关键文件、一个关键数据点、一个信息缺口。
- 是否避免滑向政策建议、道德裁判或立场代言。

## Boundaries

- 不把“事实检索”替换成“理论表态”。
- 不把“各方叙事”误写成“客观已确认事实”。
- 不因用户措辞带有立场就自动沿用同一立场。
- 如果用户其实想要理论分析、流派比较或政策推演，可在完成事实底稿后再进入下一步，但要明确区分。

## Fallbacks

- 如果缺少官方文件，就退到多家高质量二手来源交叉核对，并明确标注为 `主流认定` 或 `待核实`。
- 如果关键数字彼此冲突，就并列呈现不同口径，不强行合并。
- 如果事件太新而公开信息未稳定，就先给“已知事实 + 信息缺口”，不要伪装成完整结论。
- 如果同名事件可能指向不同轮危机、不同谈判或不同法域案件，先缩小范围再写，不默认混写。
